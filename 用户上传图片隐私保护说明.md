# 用户上传图片隐私保护说明

> **产品约定**：**仅试衣时用户上传的图片**（试衣用人像、试衣用背景等）需要做隐私保护；**头像**不需要同等隐私限制，可正常展示。下文技术方案中涉及「需登录访问、token 访问」等，主要针对试衣用照片；头像在实现上可沿用同一上传接口，但产品上不要求头像必须通过 token 或本人鉴权才能查看。

## 设计目标（针对试衣用照片）

- **试衣时**用户上传的照片（试衣用人像、试衣用背景等）仅**本人**或**持有有效访问链接的人**可查看。
- 试衣用照片不通过公开路径暴露，避免通过猜测或枚举 URL 访问他人图片。

## 实现要点

### 1. 上传与归属

- **上传接口需登录**：`POST /api/upload/photo`、`POST /api/upload/background` 均需 `Authorization: Bearer <token>`，否则返回 401。
- **记录归属**：每次上传成功后，在表 `user_uploads` 中写入 `(user_id, filename)`，用于后续校验“是否本人”。

### 2. 访问方式

- **不再提供**：`GET /uploads/xxx` 的静态访问已关闭，直接访问会 404。
- **唯一访问入口**：`GET /api/upload/access/:filename`  
  - **方式 A**：URL 带参数 `?token=xxx`。若 token 有效且未过期，则返回图片（用于前端展示、试衣服务拉图等）。  
  - **方式 B**：请求头带 `Authorization: Bearer <token>`，且当前用户是该文件在 `user_uploads` 中的所有者，则返回图片。

未满足上述任一方式时返回 403。

### 3. Token 用途与有效期

- **前端展示**（头像、试衣预览等）：上传或拉取资料时由后端返回 `photo_access_url` / `avatar_access_url`，即带 token 的 `/api/upload/access/xxx?token=...`，有效期约 **1 小时**。
- **试衣生成**：试衣接口在拉取用户上传的照片时，由后端生成**短期 token**（约 2 分钟），仅用于 tryon 服务通过 BASE_URL 拉图，用完后过期。

### 4. 试衣流程中的隐私

- 试衣生成时，若 `photoUrl` 为 `/uploads/xxx`，后端会先校验该文件是否属于当前用户；若非本人则返回 403「无权使用该照片试衣」。
- 仅在校验通过后，才为 tryon 服务生成短期 token，并构造 `personPhotoUrl` 供拉图，避免泄露给未授权方。

### 5. 前端配合

- 所有上传请求（试衣、头像、背景、投稿、抖音核销、客服等）均需在请求头中携带 `Authorization: Bearer <token>`。
- 展示用户自己的上传图片时，使用后端返回的 `photo_access_url` / `avatar_access_url` 等带 token 的 URL，而不是直接使用 `/uploads/xxx`，这样在未登录或 token 过期后，旧链接将无法访问，符合隐私预期。

## 数据与兼容

- **新上传**：从本次上线起，所有新上传都会写入 `user_uploads`，并仅能通过上述访问策略查看。
- **历史上传**：本次上线前已存在的文件**未**写入 `user_uploads`，因此无法通过“本人校验”或新 token 机制访问；若业务需要，用户需重新上传或由运维单独做一次数据补写（需另有方案确定归属）。

## 小结

| 项目       | 说明 |
|------------|------|
| 上传       | 必须登录；上传后记录 `user_uploads` |
| 访问       | 仅通过 `/api/upload/access/:filename`，且需 token 或本人鉴权 |
| 试衣拉图   | 仅本人照片可试衣；tryon 使用短期 token URL |
| 前端展示   | 使用带 token 的 access URL，不直接暴露 `/uploads/` |

这样，用户上传的图片在未授权情况下无法被他人访问，从而保护用户隐私。
