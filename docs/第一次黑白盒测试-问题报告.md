## P1-001 试衣服务无法拉取用户上传衣物图，生成结果退化为占位图

### 严重级别

- P1（高）

### 影响范围

- 影响“我的衣库”上传衣物后的真实试穿效果。
- 用户虽然能完成“生成”流程，但结果可能为占位图，影响核心体验与转化。

### 复现步骤

1. 新注册用户并登录。
2. 上传图片到 `我的衣库`（接口：`/api/upload/photo` + `/api/wardrobe`）。
3. 调用 `/api/try-on/generate`，传入 `wardrobeItemId` + `photoUrl`。
4. 观察返回体中的 `hint` 字段。

### 实际结果

- `hint` 提示试衣服务拉图失败，出现 404：
  - `http://localhost:3001/uploads/<filename>` Not Found
- 结果图回退为占位图（SVG data URL）。

### 期望结果

- 试衣服务可访问到用户上传的衣物图并正常生成真实试穿图，不应回退占位图。

### 初步原因分析

- 当前后端对上传文件主要通过 `/api/upload/access/:filename?token=...` 受控访问；
- 试衣流程中某路径仍尝试读取 `/uploads/...` 直链，而服务端未对 `/uploads` 做静态公开映射，导致 404。

### 修复建议

- 优先方案（推荐）：
  - 统一试衣侧取图入口，只使用带 token 的访问 URL（`/api/upload/access/...`），不使用 `/uploads/...` 直链。
  - 校验 `personPhotoUrl` 与衣物图 URL 两侧都走受控访问路径。
- 备选方案：
  - 新增只读静态映射 `/uploads`（有隐私风险，不推荐）。

### 验收标准

- 同一复现步骤下，`/api/try-on/generate` 返回真实图片 URL（非占位图 data URL）。
- `hint` 不再出现 404 拉图错误。
- 回归通过：原有官方衣库试穿不受影响。

### 修复进展

- 状态：**已修复并复测通过（针对 404 路径问题）**
- 证据：复测时不再出现 `/uploads/... 404`，说明衣物图 URL 已改为 token 受控访问链路。
- 备注：当前仍有独立环境问题（tryon-service 返回 502），不属于本缺陷本身，建议作为新问题单跟踪。
