## 1. 下一个功能（建议立即做）

### F1. 天气规则配置化（必做）

- **优化位置**：`backend/src/routes/recommend.ts`
- **当前问题**：天气评分规则写在代码里，后续调优要改代码、重发版。
- **优化内容**：
  - 把温度阈值、标签加减分抽成配置对象（例如 `WEATHER_SCORING_RULES`）。
  - 支持按场景微调（如“通勤”更偏向 `外套/适中`）。
- **价值**：高（调参效率提升，后续迭代成本低）。
- **开发成本**：低。
- **风险**：低。
- **本阶段建议**：**必做**。
- **验收标准**：
  - 不改业务逻辑代码，仅改配置即可调整评分结果。
  - 对同一组候选，配置修改后排序能稳定变化。

### F2. 推荐结果可解释化（必做）

- **优化位置**：后端推荐返回字段 + 前端推荐页轻提示。
- **当前问题**：用户只看到结果，不知道“为什么推荐这套”。
- **优化内容**：
  - 后端为每套返回 `weather_score_reasons`（如“低温+外套 +2”）。
  - 前端展示一句简短说明：`已按当前天气优化（降温优先外套）`。
- **价值**：高（提升信任感，便于答辩展示“可解释AI”）。
- **开发成本**：低到中。
- **风险**：低。
- **本阶段建议**：**必做**。
- **验收标准**：
  - 接口可返回可读原因列表。
  - 推荐页可看到天气优化提示文案。

## 2. 第二优先级（本阶段可做，可延后）

### F3. 标签质量治理（建议做）

- **优化位置**：`scripts/tag-outfit-images.ts` + 数据修复脚本。
- **当前问题**：标签会有噪声（例如颜色识别偏差、名称关键词误判）。
- **优化内容**：
  - 增加“黑名单标签/互斥标签”清洗（如同时出现 `轻薄` 与 `厚实` 时做优先级保留）。
  - 增加“低置信度跳过”策略（例如饱和度过低时只打中性色，不强行分色系）。
- **价值**：中高（直接影响天气重排质量）。
- **开发成本**：中。
- **风险**：低。
- **本阶段建议**：**建议做**。
- **验收标准**：
  - 抽检 50 张图，标签明显错误率下降。
  - 推荐排序“违和感”减少（主观评估可记录）。

### F4. 兜底策略精细化（建议做）

- **优化位置**：推荐路由兜底分支。
- **当前问题**：当结果很少时，兜底策略比较粗糙。
- **优化内容**：
  - 采用“逐级放宽”而不是一次放开（先弱化天气阈值，再放宽年龄，最后才放宽其他条件）。
  - 保证每次推荐数量稳定（例如 3 套）。
- **价值**：中（提升稳定性和体验）。
- **开发成本**：中。
- **风险**：低。
- **本阶段建议**：**建议做**。
- **验收标准**：
  - 常见城市/场景下推荐空结果率接近 0。
  - 推荐数量稳定。
