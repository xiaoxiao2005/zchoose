## 1. 引言 / 摘要

**Zchoose-AI 穿搭助手**是一款面向日常穿搭与虚拟试衣场景的 **Web 应用**。系统以用户个人形象为参照，结合标签化衣库（场合、季节、性别、年龄段及多元风格），提供 **穿搭推荐**、**AI 试衣（三视图）**、**我的衣库与自定义衣物试穿**、**积分与解锁**、**商家槽位与尺码匹配（基础版）**、**闲置 / 过季衣物发布与市场浏览**、**客服与工单类能力** 等功能。

软件解决的问题包括：降低「不知道穿什么」的决策成本；在用户本人照片上预览搭配效果（需配置试衣服务）；将衣库浏览、试衣、激励与闲置流转集中在同一站点，便于从「发现搭配」到「体验 / 流转」的闭环。当前以网页端为主，开发环境通过 Vite 与 Express 分别提供前端与 API，可选对接外部试衣服务与天气数据源。

## 2. 功能描述

### 2.1 主要功能模块

| 模块 | 说明 |
|------|------|
| **用户与鉴权** | 手机号验证码、注册登录；JWT 鉴权；个人资料与偏好（性别、年龄段、场合等）。 |
| **首页与导航** | 开屏 / 首页入口；底部导航与侧栏：衣库、闲置、快速穿搭、试衣、客服、我的等。 |
| **穿搭推荐** | 结合天气（可选和风 / Open-Meteo）、场合与标签从衣库筛选搭配；支持刷新与随机抽取。 |
| **衣库浏览** | 按风格、季节、性别、年龄段等多维标签筛选；卡片展示；部分搭配需积分解锁；点赞与收藏相关数据由后端表支撑。 |
| **AI 试衣** | 用户上传照片、选择官方搭配或「我的衣库」中的衣物；后端请求 `TRYON_API_URL` 生成前/侧/后三视图；未配置或上游异常时可回退占位图；等待过程提供小游戏与倒计时。 |
| **我的衣库** | 用户上传衣物图片并入库；列表与删除；与试衣流程联动；上传路径与鉴权校验防止越权与非法 URL。 |
| **积分与解锁** | 积分账户、解锁记录；投稿采纳等途径获得积分；会员档位（月 / 季 / 年）在前端展示免费试穿套数说明（后端配额计数可按迭代加强）。 |
| **商家槽位** | 搭配维度为上衣 / 裤子 / 鞋子配置商家与链接；尺码规则表支持按身高体重区间匹配示例。 |
| **闲置市场** | 用户闲置与商家过季货品发布、列表与多条件筛选（来源、标签、价格区间等）；发布页含费用说明（占位费记录）。 |
| **客服与审核** | 客服消息；指定账号可访问投稿审核、客服审核类页面（以当前路由与身份判断为准）。 |
| **支付相关** | 后端集成 Stripe 等依赖，具体收款流程以实际配置与页面为准。 |
| **隐私与合规** | 提供隐私政策页面；用户上传图片有访问控制说明与 token 化访问路径设计。 |

### 2.2 特点与问题解决方式

- **可理解的推荐**：用显式标签（场合、季节、人群）代替「黑箱」排序，用户可理解为何看到某套搭配。  
- **以用户为模特的试衣**：试衣链路围绕用户上传照片与选中衣物图，满足「看自己穿」的需求；官方衣库与用户衣库双源，兼顾内容运营与个性化。  
- **激励与商业化预留**：积分、解锁、会员说明、商家槽位与闲置市场，支撑拉新、活跃与后续交易闭环。  
- **部署灵活**：本地双端口开发；可配合 Nginx、PM2、Cloudflare Tunnel 与自定义域名进行演示或小型公网发布。

## 3. 技术特点

### 3.1 架构概述

- **前后端分离**：浏览器 SPA 通过 HTTP 调用 REST API。  
- **数据层**：SQLite 单文件数据库，通过 **sql.js** 在 Node 内加载，持久化路径默认为 `backend/data/app.db`。  
- **可选外部服务**：试衣生成 API（`TRYON_API_URL`）、天气 API。  
- **开发联调**：Vite 将 `/api`、`/uploads`、`/images` 代理到后端，避免跨域问题。

### 3.2 编程语言与主要技术

| 层次 | 技术 | 版本（以 package.json 为准） |
|------|------|------------------------------|
| 前端 | React、React Router、TypeScript、Vite | React ^18.2、Router ^6.20、TS ^5.3、Vite ^5.0 |
| 后端 | Node.js、Express、TypeScript | Express ^4.18、TS ^5.3 |
| 数据库 | sql.js（SQLite） | ^1.10.0 |
| 认证与安全 | JWT（jsonwebtoken）、bcryptjs | — |
| 上传 | multer（单文件大小限制等） | — |
| 其他 | dotenv、cors；图片处理 jimp；支付 stripe 等 | — |

### 3.3 关键工程特性

- 前端：**函数组件 + Hooks**，`AuthContext` 管理登录态，`fetch` + `Authorization: Bearer`；页面级 CSS，移动端适配。  
- 后端：路由按领域拆分（users、outfits、try-on、wardrobe、upload、resale-items、support 等）；静态资源与 `/images` 映射至 `frontend/public/images`。  
- 衣库数据可由脚本扫描 `frontend/public/images` 目录结构与后端初始化逻辑同步至 `outfits` 表。

## 5. 操作流程

### 5.1 新用户首次使用（典型路径）

1. 访问站点 → 开屏或进入首页。  
2. 在「我的」或试衣相关页完成 **手机号验证码注册 / 登录**。  
3. 在「我的」中完善 **偏好档位**（性别、年龄段等，视页面选项而定）。  
4. 浏览 **衣库** 或使用 **快速穿搭** 获取推荐；对感兴趣搭配可 **解锁**（若需积分）。  
5. 进入 **试衣**：上传合规本人照、选择官方搭配或 **我的衣库** 中衣物 → 生成三视图 → 查看 / 下载（视功能开放情况）。  
6. 可选：在 **闲置市场** -> **发布** 填写信息上架衣物；在 **客服** 发起咨询。

### 5.2 衣库与试衣（含我的衣库）

1. **我的** → **我的衣库** → 上传衣物图 → 通过前端质量提示与后端校验。  
2. **试衣** → 切换衣库来源 → 选择衣物与姿态 → 等待生成 → 若上游服务不可用，可能看到占位图与错误提示。

### 5.3 管理员 / 审核员（若启用）

1. 使用配置中具备审核权限的账号登录。  
2. 访问 `/submission-review` 或 `/support-review` 处理投稿与客服工单。

## 6. 安装和部署

### 6.1 开发环境安装（Windows / macOS / Linux 通用）

**前置**：Node.js **18+**、npm。

1. **后端**  
   ```bash
   cd backend
   npm install
   cp .env.example .env
   # 编辑 .env：PORT、DATABASE_PATH、JWT_SECRET、TRYON_API_URL（可选）、天气相关变量（可选）
   npm run dev
   ```
   默认监听 `http://localhost:3001`（以 `.env` 为准）。

2. **前端**  
   ```bash
   cd frontend
   npm install
   npm run dev
   ```
   默认 `http://localhost:5173`，通过 Vite 代理访问后端。

### 6.2 生产构建与部署要点

1. 前端：`npm run build`，产物在 `frontend/dist`。  
2. 后端：`npm run build` → `npm run start` 或使用 **PM2** 运行 `node dist/index.js`。  
3. 推荐使用 **Nginx** 托管 `dist` 并将 `/api`、`/uploads` 等反代至本机后端端口；**HTTPS**、**强 JWT_SECRET**、**不直接暴露** `.env` 与数据库文件。  
4. 可选：**Cloudflare Tunnel** 将本机服务映射到 `https://你的域名`（参见项目内 `Cloudflare-Tunnel-使用说明.md`、`部署与安全说明.md`）。

### 6.3 数据与资源

- 首次启动或迁移时注意 `backend/data/app.db` 与 `backend/uploads/` 的权限与备份。  
- 衣库图片资源位于 `frontend/public/images`，需与后端扫描/初始化逻辑一致。

## 8. 测试结果

本项目已进行 **第一轮黑白盒测试**（详见 `docs/第一次黑白盒测试-测试方法与执行记录.md` 与问题报告），摘要如下。

### 8.1 测试范围与方法

- **黑盒**：注册 → 上传 → 我的衣库 → 列表 → 试衣生成全链路接口。  
- **白盒**：非法图片 URL 入库、无效 token、缺少试衣参数、越权试穿 / 删除等分支。  
- **环境**：Windows 10、后端 `localhost:3001`、`TRYON_API_URL` 已配置、SQLite 开发库。

### 8.2 结果摘要

| 类型 | 结论 |
|------|------|
| 黑盒主流程 | `send-code`、`register`、`upload/photo`、衣库写入/查询、`try-on/generate` 等关键接口可按预期（200/201）跑通。 |
| 白盒安全分支 | 非授权路径、无效 token、参数缺失、越权试穿与删除等用例行为符合设计（400/401/403/404 等）。 |
| 已知依赖 | 试衣真实效果图依赖上游 tryon-service 可用性；上游异常时可能回退占位图。 |
| 缺陷闭环 | 用户上传图在试衣链路中的受控 URL（token）问题已按记录修复；环境类 502 等需运维侧排查。 |

### 8.3 性能指标

未在全文档中固定公布统一 **SLA 数值**；建议在后续版本中针对「列表分页」「并发试衣」「大库扫描」补充压测报告与目标指标（如 P95 延迟、错误率）。
