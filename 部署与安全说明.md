# 公网部署与安全说明

## 一、安全吗？怎么做才安全

**直接暴露在公网而不做任何配置，是不安全的。** 按下面做法配置后，可以做到**相对安全**，适合个人项目或小规模使用。

### 1. 必须做的安全措施

| 措施 | 说明 |
|------|------|
| **HTTPS** | 用 Nginx + Let's Encrypt 免费证书，避免密码、token 在网络上明文传输。 |
| **强 JWT 密钥** | 生产环境 `.env` 里 `JWT_SECRET` 改为随机长字符串（如 32 位以上），且不要提交到 Git。 |
| **不暴露敏感文件** | `.env`、`data/app.db`、`uploads/` 只能由服务器进程访问，不要通过 Nginx 直接对外提供。 |
| **防火墙** | 只开放 80、443，后端 3001 只监听 127.0.0.1，不对外网开放。 |
| **环境变量** | 所有 API 密钥、数据库路径、BASE_URL 等只写在服务器上的 `.env`，不写进代码。 |

### 2. 建议做的

- **定期备份**：备份 `backend/data/app.db` 和 `backend/uploads`。
- **强密码**：若开放注册，可加验证码、限流，避免暴力破解。
- **更新依赖**：`npm audit` 查漏洞，定期 `npm update`。

## 二、怎么实现：部署到有公网的服务器

以 **一台 Linux 云服务器（阿里云 / 腾讯云 / 其他 VPS）+ 域名（可选）** 为例。

### 2.1 准备

- 一台有公网 IP 的 Linux 服务器（如 Ubuntu 22.04）。
- （可选）域名解析到该公网 IP，便于后续上 HTTPS。
- 本机已安装 Node.js 18+、npm。

### 2.2 服务器上安装环境

```bash
# 以 Ubuntu 为例
sudo apt update
sudo apt install -y nodejs npm nginx
# Node 建议用 nvm 装 18+：curl -o- https://raw.githubusercontent.com/nvm-sh/nvm/v0.39.0/install.sh | bash
```

### 2.3 上传代码并构建

在**本机**打包前端、上传到服务器（或用 Git 在服务器拉取）：

```bash
# 本机：前端打包
cd frontend
npm install
npm run build
# 得到 frontend/dist 目录
```

把整个项目（或至少 `backend` + `frontend/dist`）上传到服务器，例如放到 `/var/www/zchoose`：

- `backend/`：后端代码与 `data/`、`uploads/`
- `frontend/dist/`：前端构建结果（或整个 frontend 在服务器上执行一次 `npm run build`）

### 2.4 后端以生产方式运行

在服务器上：

```bash
cd /var/www/zchoose/backend
npm install --production
# 复制并编辑 .env，务必改 JWT_SECRET、BASE_URL（见下）
cp .env.example .env
nano .env
npm run build
node dist/index.js
```

建议用 **PM2** 保活与重启：

```bash
npm install -g pm2
pm2 start dist/index.js --name zchoose-api
pm2 save
pm2 startup
```

**生产环境 `.env` 关键项：**

```env
NODE_ENV=production
PORT=3001
JWT_SECRET=请改为随机长字符串至少32位
BASE_URL=https://你的域名
# 若用域名：BASE_URL=https://your-domain.com
# 若只用 IP：BASE_URL=http://你的公网IP
DATABASE_PATH=./data/app.db
```

### 2.5 Nginx 反向代理（同一台机推荐）

这样用户只访问 80/443，Nginx 把请求转给前端静态和后端 API，**后端 3001 只对本机开放，更安全**。

创建站点配置（若用域名，把 `your-domain.com` 换成你的域名或直接用服务器公网 IP 作 server_name 测）：

```bash
sudo nano /etc/nginx/sites-available/zchoose
```

写入（按需改 `your-domain.com`、`root` 路径）：

```nginx
server {
    listen 80;
    server_name your-domain.com;   # 没有域名就写 服务器公网IP 或 _

    # 前端静态（Vite 打包后的 dist）
    root /var/www/zchoose/frontend/dist;
    index index.html;
    location / {
        try_files $uri $uri/ /index.html;
    }

    # 后端 API 与静态资源转发到本机 3001
    location /api/ {
        proxy_pass http://127.0.0.1:3001;
        proxy_http_version 1.1;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
    location /uploads/ {
        proxy_pass http://127.0.0.1:3001;
        proxy_set_header Host $host;
    }
    location /images/ {
        proxy_pass http://127.0.0.1:3001;
        proxy_set_header Host $host;
    }
}
```

启用并重载 Nginx：

```bash
sudo ln -s /etc/nginx/sites-available/zchoose /etc/nginx/sites-enabled/
sudo nginx -t
sudo systemctl reload nginx
```

此时用 **http://你的域名或公网IP** 即可访问（未配 HTTPS 前是明文，仅适合测试）。

### 2.6 配置 HTTPS（推荐，更安全）

有域名时，用 Let's Encrypt 免费证书：

```bash
sudo apt install certbot python3-certbot-nginx
sudo certbot --nginx -d your-domain.com
```

按提示选证书和自动跳转 HTTPS。完成后访问 **https://你的域名**。

**注意**：配置好 HTTPS 后，把 `backend/.env` 里 `BASE_URL` 改为 `https://你的域名`，并重启后端（如 `pm2 restart zchoose-api`）。

### 2.7 前端请求的「当前域名」

前端若用相对路径请求（如 `/api/xxx`、`/uploads/xxx`），部署到同一域名下时无需改代码，Nginx 会按上面的 `location` 转发。若你之前写死了 `localhost`，需要改成相对路径或通过环境变量在构建时注入 API 地址（Vite 可用 `import.meta.env.VITE_API_BASE` 等）。

## 三、线上登录不上（如 https://z-choose.com/）排查

访问 https://z-choose.com/ 时「登录不上」通常是因为：**前端页面能打开，但登录请求 `/api/users/login` 没有到达后端**，或后端未正确响应。按下面步骤排查。

### 3.1 确认登录请求发往哪里、返回什么

1. 打开 **https://z-choose.com/**，按 F12 打开开发者工具，切到 **Network（网络）** 面板。
2. 勾选 **Preserve log**（保留日志），在登录框输入手机号、密码，点击登录。
3. 在列表里找到 **`login`** 或 **`users/login`** 的请求，点开查看：
   - **Request URL**：应为 `https://z-choose.com/api/users/login`（与页面同域）。
   - **Status（状态码）**：
     - **404** → 说明当前域名下没有把 `/api` 转到后端，见 3.2。
     - **502 / 503** → 后端未启动或 Nginx 转发的地址/端口不对，见 3.3。
     - **0 或 CORS 报错** → 请求被浏览器拦截（跨域），见 3.4。
     - **200** → 接口通了，若仍提示登录失败，看 Response 里返回的 `error` 文案（如账号密码错误、验证码错误等）。

### 3.2 状态 404：后端未接入当前域名

本应用前端使用**相对路径** `/api/xxx`，即：**谁提供页面，谁就要同时提供 `/api`**。

- 若 **只部署了前端**（例如只把 `frontend/dist` 放到 Vercel / Netlify / 对象存储），没有在同一域名下部署 Node 后端，则 `https://z-choose.com/api/users/login` 会 404，登录必然失败。
- **正确做法**：在 **z-choose.com 所在服务器**上同时跑 Node 后端，并用 Nginx 把 `/api`、`/uploads`、`/images` 反向代理到该后端（见上文 2.5 的 Nginx 配置）。  
  即：**同一台机（或同一域名）上既有前端静态（dist），又有后端（Node 监听 3001），Nginx 把 /api 转给 3001**。

### 3.3 状态 502/503：后端未运行或 Nginx 配置错误

- 在服务器上确认后端已启动并监听 3001：  
  `pm2 list` 或 `curl -s http://127.0.0.1:3001/api/health` 能返回正常结果。
- 确认 Nginx 里 `proxy_pass http://127.0.0.1:3001;` 的端口与后端一致，且没有多写或少写斜杠（如 `proxy_pass http://127.0.0.1:3001/;` 会改路径，一般用无尾斜杠）。

### 3.4 其他（0 / CORS / 控制台报错）

- 若前端和接口**不在同一域名**（例如页面在 z-choose.com，接口在 api.z-choose.com 且未在 Nginx 里用 `/api` 转发），需在后端配置 CORS 允许前端域名，或改为在同一域名下用 Nginx 转发 `/api`（推荐）。
- 若控制台有 **Mixed Content** 报错：页面是 HTTPS，请求被发到 HTTP，也会失败；确保 Nginx 与 `BASE_URL` 均为 HTTPS。

### 3.5 小结

| 现象 | 可能原因 | 处理 |
|------|----------|------|
| 登录接口 404 | 仅部署了前端，或 Nginx 未把 `/api` 转给后端 | 在同一域名下部署 Node 后端，并配置 Nginx 反向代理 `/api`、`/uploads`、`/images` |
| 登录接口 502/503 | 后端未启动或 Nginx 转发地址/端口错误 | 启动后端（如 PM2），检查 Nginx `proxy_pass` |
| 接口 200 仍提示失败 | 业务错误（账号/密码/验证码等） | 看接口返回的 `error` 字段，按提示修正 |
| 请求被跨域/CORS 拦截 | 前后端不同域且未正确配置 | 尽量同域 + Nginx 转发；或后端配置 CORS |

## 四、简要检查清单

部署后可按此自检：

- [ ] 仅 80/443 对公网开放，3001 仅 127.0.0.1
- [ ] `.env` 中 `JWT_SECRET` 已改为强随机串，`BASE_URL` 为实际访问地址（最好 HTTPS，如 `https://z-choose.com`）
- [ ] **Nginx 已把 `/api`、`/uploads`、`/images` 反向代理到本机 3001**（否则线上登录会 404）
- [ ] 能用 **https://你的域名**（或 http，若未配证书）正常打开页面、**登录**、试衣
- [ ] 数据库与上传目录不在 Nginx 里直接暴露（只通过后端接口访问）

按上述步骤做，部署到公网服务器并配合 HTTPS 与强密钥，**安全性可以满足个人或小规模使用**；若后续要承载更多用户，可再考虑 WAF、限流、数据库独立与备份策略等。
