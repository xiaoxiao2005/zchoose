# 五、商业模式

## 本章导读（总览）

**Zchoose** 面向城市通勤人群为主力 C 端用户，以「场景化推荐 + 真人试穿确定性 + 衣库与商家槽位 + 闲置/过季流转」形成差异化闭环；价值上侧重**决策效率**与**购前确定性**，辅以积分、会员在「积分解锁类搭配」上的权益设计。收入上，**C 端会员订阅已接入在线支付（Stripe Checkout）**作为可验证的增值来源；**B 端槽位费、闲置占位费、交易佣金等以规则先行、分阶段实扣与对账为准**。成本上，**外部试衣 API 按调用计费**为刚性增长项，须与时尚能量等用量规则及中长期 B 端分成协同，保证单位经济性与可持续运营。

本章说明面向谁创造价值、价值是什么、通过哪些渠道触达用户、客户关系如何维护、成本与收入如何设计、关键合作方向，以及**验证进展、阶段假设与风险边界**。当前阶段以产品验证与小范围运营为主；**完整财务报表与多年预测可单列「财务预测」章节或附录**，本章不展开未经说明的具体数字。

## （〇）验证进展与商业化阶段（建议答辩时口头强调）

**表 5-0 验证与阶段（定性）**

| 阶段 | 重点目标 | 当前进展（与系统实现为准） |
|------|----------|----------------------------|
| 验证期 | 核心闭环可用、留存可观测、付费意愿可测 | 衣库/试穿/积分/闲置/客服等主路径已贯通；会员档位与**在线支付开通链路**已落地；小范围试用与演示可开展 |
| 增长期 | 扩大商家触达与衣库供给，C 端会员与 B 端合作并行放量 | 以试点与可复盘数据为前提，避免写成「已大规模入驻」 |
| 成熟期 | 交易闭环、分成结算、数据增值服务（合规前提下） | 依赖合同与系统能力，适合中后期单独规划 |

**尚未验证或不夸大表述**：大规模商家入驻、成交佣金自动分账、闲置占位费全面实扣等，均可在答辩中明确列为「下一阶段验证项」。

## （一）客户细分

项目同时服务 C 端用户与 B 端商家，并连接闲置/过季供给方（个人与商家）。细分不追求「概念很多」，而追求场景清楚、付费逻辑说得通。

1. **C 端：城市打工人（主力人群）**  
   典型特征：通勤与职场场景多、时间碎片化；希望少花时间完成「今天穿什么」的决策；对「先看自己穿上如何」的试穿方式接受度高；对理性购衣、减少闲置有真实需求。  
   **需求重点**：场景化推荐、当季组织、试穿确定性、购买路径清晰、闲置可处理。  

   旧稿中的「兴趣用户（星座/古装/二次元等）」不作为当前主线细分；若未来做内容扩展，可作为增长型细分单独写进路线图，避免冲淡「打工人场景化」主定位。

2. **C 端：高活跃/高贡献用户（规模小但价值高）**  
   典型特征：更愿意投稿互动、愿意分享内容，对会员权益与积分规则更敏感。  
   **价值**：提升社区供给与传播，但不等于「亚文化人群」才是核心用户。

3. **B 端：服饰商家（合作方）**  
   典型特征：需要曝光与转化；希望以较低门槛获得「搭配场景下的精准展示」；存在过季库存清理需求。

4. **供给方：闲置卖家与过季发布方**  
   个人：希望闲置衣物在穿搭属性更清晰的语境里流转。  
   商家：希望过季商品有独立展示与筛选路径（与衣库标签体系一致，便于与大厅场景形成差异）。

## （二）价值主张

**表 5-1 价值主张（对齐当前产品）**

| 价值维度 | 核心内容（用一句话说清） | 主要面向 |
|----------|--------------------------|----------|
| 决策效率 | 先按场合/季节/人群缩小范围，再进入衣库与试穿，减少「从海量内容里捞答案」的成本 | 打工人用户 |
| 确定性 | 以用户为模特的试穿（外部试衣服务），先看效果再决定，降低「买回来不合适」的心理与金钱成本 | 打工人用户 |
| 交易辅助 | 商家槽位展示上衣/裤子/鞋子/配饰等商品信息，帮助从「看搭配」到「找单品」 | 用户与商家 |
| 价格与流通 | 闲置与过季专区提供额外选择与去库存路径；发布门槛设计兼顾治理成本（以页面规则为准） | 闲置卖家、商家 |
| 轻量激励 | 积分、解锁、会员在积分解锁类搭配上的免费解锁配额等（以系统规则为准） | 活跃用户 |

**与常见竞品的差异（商业模式侧，半页以内）**：  
- 相对「纯试衣工具」：多衣库场景化组织、商家槽位与闲置专区，离「决策—购买—流转」更近。  
- 相对「纯导购/电商」：强调以用户为主体的试穿确定性，而非仅商品图与文案。  
- 相对「纯社区」：激励与解锁规则与衣库、试穿路径绑定，避免只做内容没有闭环。

## （三）渠道通路

1. **线上自有渠道（当前主阵地）**  
   Web 端：承载衣库、试穿、闲置、客服与个人中心等核心功能。对外演示与长期使用建议使用可访问的正式域名与环境（具体域名与部署以团队实际为准），避免在计划书中把本地开发地址写为主渠道。  
   移动端：以手机网页适配为重点（符合通勤场景），App/小程序可作为后续选项，不作为当前已完成承诺。

2. **内容与增长渠道（探索中，强调方法迭代）**  
   短视频/内容平台：团队可进行话题类尝试（例如围绕「今天穿什么不再难」等），当前更关注不仅曝光、还要能复盘（与第三章指标表一致）。  
   商家拓展：可在阿里巴巴 1688 等渠道寻找愿意提供商品信息与素材的商家合作，但需写清处于**触达与试点阶段**，避免写成「已大规模入驻」。

3. **口碑与裂变（轻量、可控）**  
   投稿奖励、分享传播等机制可以写，但须补一句：**以合规与内容治理为前提**，避免夸大「自发裂变一定爆发」。

## （四）客户关系

1. **自助服务（主路径）**  
   推荐、衣库筛选、试穿、闲置浏览与发布、会员开通与支付入口等以自助为主，降低服务成本。

2. **人工服务（辅助路径）**  
   客服入口用于规则咨询、反馈收集、异常处理；是否包含投稿处理、核销信息提交等，以系统实际功能为准。

3. **协作共创（轻量）**  
   用户投稿与反馈用于优化衣库内容与推荐策略；这是「可持续运营」的关键，但须用**采纳规则**写清楚，避免空话。

## （五）成本结构与收入来源

### 1. 收入与成本的对应关系（逻辑闭环）

- **试衣 API 成本**：外部试衣服务通常**按次计费**，随调用量线性上升，是需重点监控的刚性成本。  
- **用量控制**：通过时尚能量、每日试衣额度等规则，避免无上限免费调用导致成本失控（详见《激励制度与盈利方案-单独文档》）。  
- **收入来源分工**：中长期以 **B 端合作（引流、按量或保底）** 覆盖试衣与运营的主要增量；**C 端会员**作为已落地的增值，侧重「积分解锁类搭配」的当期免费解锁额度，与「按张售卖试衣」区分；闲置占位费、交易佣金等适合在**治理与闭环成熟后**分阶段打开。  
- **一句话**：**先控量、再谈规模；B 端与规则为主轴，C 端会员为补充，避免单一依赖补贴或单一收费项。**

### 2. 成本结构

**表 5-2 成本结构**

| 成本项 | 说明 | 现阶段特征 |
|--------|------|------------|
| 研发成本 | 产品迭代、测试与维护 | 团队自研为主 |
| 基础设施 | 域名、服务器、隧道/网关等 | 小规模可控 |
| 试衣服务成本 | 外部试衣 API 通常按调用计费 | 随使用量上升，需重点监控 |
| 内容与运营 | 衣库上新、审核、客服、活动 | 随规模上升 |
| 获客与推广 | 内容制作、投放（如有） | 建议小步试验、强复盘 |
| 支付与合规 | 第三方支付通道、对账与合规适配 | 随 C 端付费与 B 端结算落地而增加 |

### 3. 收入来源

下列收费标准以当前产品页面与规则为参考；**会员费已具备真实支付与开通能力**；其余项目以「规则先行、分阶段实扣与对账」为原则。答辩口径可概括为：**「会员商业化已技术闭环；其余收费项分阶段推进。」**

**表 5-3 收入来源（设计与分阶段落地）**

| 收入类型 | 收费对象 | 收费标准/模式（参考现有说明） | 说明 |
|----------|----------|-------------------------------|------|
| 会员费 | C 端用户 | 页面档位：月/季/年（如 10 / 35 / 100 元）及当期「积分解锁类搭配」免费解锁次数说明 | **已接入 Stripe Checkout**；支付成功延长有效期并发放当期额度；Webhook 与支付成功回跳补单；开发环境可模拟支付 |
| 商家槽位费 | B 端商家 | 页面规则示例：29.9 元/月/槽位（文案中常提示尚未接入实际扣费与结算） | 产品化收费方向清晰；自动结算与合同需分阶段 |
| 闲置发布费 | 个人/商家 | 占位费设计（如 2 元/套），可能标注暂不实际扣费 | 治理恶意发布；落地需审核与申诉配套 |
| 交易佣金 | B 端商家 | 成交分成（比例与合作方式绑定） | 依赖交易闭环与合同体系，适合中后期 |
| 数据增值服务 | B 端 | 趋势洞察、尺码合作等 | 须在合规与授权前提下；当前以 BMI 等轻量信息为主，商家尺码未接入前不夸大 |

### 4. 未来 12～24 个月商业化节奏（定性里程碑，无具体假数字）

| 时间维度 | 方向性目标（可随校赛/国赛节点调整） |
|----------|--------------------------------------|
| 0～6 个月 | 巩固主路径留存；积累会员付费与试用数据；小规模内容/商家试点并复盘 |
| 6～12 个月 | 在数据支撑下，明确是否加大 B 端试点或槽位费试点；完善闲置治理与申诉 |
| 12～24 个月 | 视闭环与合同能力，探索交易佣金或区域化商家合作；数据增值服务仅在合规前提下 |

## （六）重要合作

1. **供给与合作**  
   服饰商家：提供可展示的商品信息与链接，丰富槽位与过季专区。  
   内容与合作方：以「可持续供给」为目标，优先解决打工人场景衣库扩充与质量稳定。  
   旧稿中「设计师/非遗/民族服饰深度绑定」等表述，建议改为**可选的社会价值延伸方向**，避免与当前主线不一致。

2. **技术合作**  
   试衣服务提供商：保障试衣链稳定与成本可控；保留可替换空间。  
   云与基础设施：保障访问稳定与演示可用。  
   支付服务（如 Stripe）：保障收银台、回调与合规配置；密钥与 Webhook 须按环境管理。

3. **渠道合作（务实写法）**  
   与抖音等内容平台的关系建议写成：**先做可复盘的小实验，再决定是否加码**，而不是「必然引流爆发」。

## （七）社会价值（与国家战略/绿色消费表述衔接，宜短）

- **理性购衣与减少浪费**：先试后决策、闲置与过季专区促进衣物复用，契合绿色消费与节约型社会倡导（表述宜与真实功能一致，不夸大）。  
- **就业与小微商家**：以低门槛展示与试点合作，服务中小服饰商家曝光与去库存需求（分阶段，不承诺必然规模）。  
- **文化展示（可选延伸）**：衣库多元风格可作为体验与传播入口；若涉及真实民族服饰或非遗合作，须单独论证授权与合规，不作为当前主线承诺。

## （八）关键假设与边界

| 类型 | 假设或边界 | 说明 |
|------|------------|------|
| 外部服务 | 试衣 API 可用、价格与条款可能变动 | 需监控成本与备选方案（如替换引擎） |
| 支付与合规 | 第三方支付通道与服务地监管要求 | 密钥、隐私、跨境与税务按规范执行 |
| 商家与交易 | B 端大规模成交分成依赖闭环与合同 | 未闭环前不写「已规模化变现」 |
| 数据与隐私 | 用户数据与洞察服务须在授权与合规前提下 | 商家尺码等未接入前不夸大「数据产品」 |

## （九）商业模式相关风险与应对（简表）

| 风险 | 应对思路（方向性） |
|------|---------------------|
| 试衣成本上升或 API 调价 | 用量规则与产品结构联动；评估 B 端分摊与会员档位；保留技术备选 |
| 用户付费意愿不足 | 先验证权益感知与留存；小步调价与权益组合测试；不依赖单一收费项 |
| 商家拓展慢于预期 | 试点案例与可复盘数据；降低试点门槛；与营销章节「1688/社媒」协同 |
| 支付政策与合规变化 | 关注通道规则；预留合规与对账能力；计划书与答辩如实披露阶段 |
| 竞争与替代（通用 AI 等） | 强化场景化衣库 + 试穿 + 槽位/闲置的闭环差异，避免单点「比效果图」 |
